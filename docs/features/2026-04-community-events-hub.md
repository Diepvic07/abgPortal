## Problem Statement

ABG Alumni Connect has 500+ members scattered across the globe. The goal is an OMO (Offline Merged Online) community where members support each other's career and life growth. Currently, event announcements are fragmented across Zalo, Facebook, email, and the website with no single source of truth. Members have platform fatigue — they mute Zalo groups, don't check Facebook. There's no shared space for discussion, no way to RSVP to events, and no lifecycle connecting proposals (ideas) to events (execution) to outcomes (impact).

## Demand Evidence

- 500+ members, 100 active, globally distributed Vietnamese alumni
- Members actively mute Zalo/Facebook groups (platform fatigue, not disinterest)
- 20% of members already propose activities to the Board through informal channels
- Proposals system shipped and validated: members engage with commitment-weighted voting
- The gap: members can propose activities but there's no path from "approved proposal" to "event with participants"
- Members want to "discuss and show up, not just be aware"

## Status Quo

Events are announced via a mix of Zalo groups, Facebook posts, email blasts, and static news articles on the website. No single channel covers all members. Members who are interested can't RSVP, can't discuss the event, and can't see who else is going. The Board has no data on expected attendance. After events, there's no record of who participated or what impact it had.

The proposals system created a pipeline of community ideas but has no output — approved proposals don't become events. The lifecycle is broken: propose → approve → ???.

## Target User and Narrowest Wedge

**Primary user:** Active ABG members who want to know what's happening, discuss events with other members, and commit to participating.

**Narrowest wedge:** A Community Hub at /community with three tabs (Proposals, Events, Completed). Events can be created from approved proposals (inheriting content and committed members) or directly by admins. Each event has RSVP with commitment level, a discussion thread, and attendance tracking. Member profiles show a contribution/commitment score.

## Constraints

- Existing stack: Next.js 16, Supabase, Vercel, Tailwind CSS 4
- Must integrate with existing auth (NextAuth, Google OAuth + magic link)
- Must work with existing member approval flow
- Must coexist with existing proposals system (proposals link to events, not replaced)
- Vietnamese alumni community, verified members, high-trust environment
- i18n required (Vietnamese + English)
- Web-only platform (no mobile app)
- PostHog analytics integration

## Premises

1. The solution lives inside ABG Connect (the existing web app), not a third-party platform
2. Notifications are the hook, discussion threads are the value — but notifications (web push + email digest) are Phase 2, not Phase 1
3. Phase 1 focuses on Events + Community Hub + the proposal-to-event lifecycle
4. Per-event discussion threads (async, structured), not group chat
5. RSVP with commitment level (reuse the proposals pattern)
6. Proposals that are approved by admin can link to events — content becomes event description, committed members become participants
7. Commitment/contribution scores visible on member profiles (reputation system)
8. The existing proposals system coexists; proposals feed into events, not replaced

## Cross-Model Perspective

Independent Claude subagent review (cold read, no conversation context):

**Steelman:** ABG is building the single digital home for a diaspora alumni network — a place where scattered members stop context-switching between Zalo/Facebook/email and instead gravitate to one platform that combines async discussion, event coordination with skin-in-the-game commitment, and AI that surfaces the right people and conversations to each member automatically. The bet is that relevance (not reach) solves platform fatigue.

**Key insight:** "We want members to discuss and show up, not just be aware" is the whole product in nine words. It draws a hard line between broadcast (what they have now) and participation (what they need). The core metric isn't DAU — it's commitment converted to action.

**Challenged premise:** Web push notifications may be wasted engineering for a community that mutes notifications. The problem was never email as a channel — it was email as broadcast. A personalized AI-picked digest is the opposite of "boring one-way email." User chose to keep web push in the plan but defer to Phase 2, focusing Phase 1 on building the destination (community hub) first.

**Prototype recommendation:** community_threads + replies + commitments. AI picks top 10 relevant members per thread and emails them. Test whether AI-targeted relevance + commitment mechanics make people discuss and show up.

## Approaches Considered

### Approach A: "Enhanced News" — Bolt events onto existing articles
Add RSVP + comments to news articles with category "Event". Reuse proposal patterns. No new entity.
- Effort: S (human ~3 days / CC ~20 min)
- Risk: Low
- Completeness: 4/10
- Rejected: Events and articles are fundamentally different entities. Shoehorning creates technical debt and blocks the community hub.

### Approach B: "Events + Community Hub" (CHOSEN)
New events table with proper fields. Event comments + commitments. /community as the unified feed. Proposal-to-event lifecycle.
- Effort: M (human ~1 week / CC ~30 min)
- Risk: Medium
- Completeness: 7/10

### Approach C: "Community Threads" — Generalized thread model
Single community_threads table with type field. Most extensible but upfront cost higher than needed for Phase 1.
- Effort: M-L (human ~1.5 weeks / CC ~40 min)
- Risk: Medium
- Completeness: 9/10
- Deferred: elegant architecture but paying upfront for flexibility not needed yet.

## Recommended Approach

**Approach B: Events + Community Hub** because it gives proper event entities without overengineering. The proposal-to-event lifecycle is the key innovation — it closes the loop from "idea" to "execution" to "outcome." When discussions and Q&A are needed (Phase 3), the system can be generalized with real usage data.

#### 2. Events System

**Event entity (new table: `community_events`):**
- id, title, description, category
- event_date, event_end_date (optional), location, location_url (optional)
- capacity (optional), image_url
- created_by_member_id, proposal_id (nullable — links to source proposal)
- status: draft, published, cancelled, completed
- rsvp_count (denormalized), comment_count (denormalized)
- created_at, updated_at, published_at, completed_at

**Note on "approved members":** Throughout this document, "approved member" means a member with `status = 'active'` in the members table (i.e., passed the existing member approval flow).

**Event creation paths:**
1. **From proposal:** Admin approves proposal → "Create Event from Proposal" action
   - Title, description, category auto-populated from proposal
   - All committed members from proposal become event RSVPs (same commitment level)
   - Uses `INSERT ... ON CONFLICT (event_id, member_id) DO NOTHING` to handle edge cases where a member RSVP'd independently
   - proposal_id links the event back to its source
   - Proposal status changes to "selected" or "in_progress"
   - Event is created with status `draft`. Admin fills in date/location, then publishes when ready.
2. **Direct by admin:** Admin creates event from scratch in /admin
   - Same fields, no proposal link
   - Event is created with status `draft`. Admin publishes when ready.

**Proposal detail page:** When a proposal has a linked event (via proposal_id on the event), the proposal detail page shows a "View Event" link to the event page.

**Event RSVP (new table: `community_event_rsvps`):**
- id, event_id, member_id
- commitment_level: interested (0 points), will_participate (3 points), will_lead (5 points)
- Same commitment model as proposals
- actual_attendance (nullable, boolean — admin marks after event)
- actual_participation_score (nullable, 1-10 — admin rates after event)
- created_at, updated_at
- UNIQUE(event_id, member_id)

**Event comments (new table: `community_event_comments`):**
- id, event_id, member_id, body, status (visible/hidden/removed)
- created_at, updated_at
- Same pattern as proposal comments

#### 3. Proposal-to-Event Lifecycle

```
PROPOSAL LIFECYCLE (existing + new transitions):
  published → selected → in_progress → completed
                ↓
          CREATE EVENT (new)
            - Admin action: "Create Event from Proposal"
            - Copies content, imports committed members
            - Links via proposal_id

EVENT LIFECYCLE:
  draft → published → completed
    ↓
  cancelled

FULL FLOW:
  Member proposes → Others commit → Admin approves →
  Admin creates event from proposal → Members RSVP →
  Event happens → Admin records attendance →
  Commitment scores update → Event moves to Completed tab
```

#### 4. Member Commitment Score (profile enhancement)

Visible on member profiles. Calculated from:
- Proposals created (published): +5 per proposal
- Proposal commitments: +commitment_level points per proposal
- Event RSVPs: +commitment_level points per event
- Event attendance (actual): +5 bonus for showing up
- Event participation score: +actual_participation_score from admin rating

Display: total score + breakdown (proposals, events, attendance)

Implementation: denormalized `commitment_score` column on `members` table, updated via trigger or cron.

#### 5. Discussion Threads on Events

Same pattern as proposal comments:
- Flat comments (no nesting — at 100 active members, nesting adds UI complexity for zero value)
- Named comments with member avatar
- Author can edit/delete own
- Admin can moderate (hide/remove)
- Comment count displayed on event card in hub

### Denormalized Field Sync

- `rsvp_count`, `rsvp_score`, `comment_count` on `community_events` kept in sync via Postgres triggers on INSERT/UPDATE/DELETE of `community_event_rsvps` and `community_event_comments`
- `commitment_score` on `members` recalculated via a **scheduled cron job** (hourly or on-demand via admin action). Multi-table aggregation across proposals + events + RSVPs + attendance makes triggers too complex for this field. Cron is simpler and sufficient at this scale.
- Same trigger pattern as existing proposal triggers for rsvp_count/rsvp_score/comment_count

### Commitment Weight Function

Reuse existing pattern:
```sql
CREATE OR REPLACE FUNCTION event_commitment_weight(level text)
RETURNS int AS $$
BEGIN
  RETURN CASE level
    WHEN 'will_lead' THEN 5
    WHEN 'will_participate' THEN 3
    WHEN 'interested' THEN 0
    ELSE 0
  END;
END;
$$ LANGUAGE plpgsql IMMUTABLE;
```

### Event Status State Machine

```
  draft → published → completed
    ↓         ↓
  cancelled  cancelled
```

- Only admins can change status
- draft: event created but not visible to members. Initial status for all new events.
- published: visible, members can RSVP and comment
- completed: event is over, moved to Completed tab, admin can record attendance
- cancelled: terminal, reachable from both draft and published. Cancelled events retain RSVPs for record-keeping. Commitment scores from cancelled events are excluded from calculation. Cancelled events are visible with a "Cancelled" badge but hidden from the default Events tab (shown only via filter). RSVPs notified via email (Phase 2).

**No hard delete:** Admins cannot delete events. Use `cancelled` status instead. This preserves RSVPs, comments, and audit trail.

### Proposal-to-Event Import Logic

When admin creates event from proposal:
1. Copy proposal title, description, category to event fields
2. Set proposal_id on event
3. For each commitment on the proposal:
   - Map commitment_level: will_lead → will_lead, will_participate → will_participate, interested → interested
   - Create RSVP row for the member with same level
4. Update proposal status to "selected" (or "in_progress" if admin chooses)
5. Update rsvp_count and rsvp_score on event via trigger
6. Admin fills in: event_date, location, capacity, image

### Member Profile Commitment Score Display

On member profile page, show:
```
Commitment Score: 152

Breakdown:
- Proposals created: 3 (15 pts)
- Proposal commitments: 8 proposals (42 pts)
- Event RSVPs: 12 events (50 pts)
- Events attended: 10 (50 pts bonus)
- Participation ratings: total 45 pts (from 6 rated events)
```

Score formula:
```
total = (proposals_created * 5)
      + SUM(proposal_commitment_weights)
      + SUM(event_rsvp_commitment_weights)
      + (events_actually_attended * 5)
      + SUM(actual_participation_scores)
```

Note: Only non-cancelled events count toward the score. Commitment scores are recalculated via a **scheduled cron job** (hourly or on-demand), not via triggers. At 100 active members, a multi-table aggregation on every write is unnecessary complexity. A cron-based approach is simpler and sufficient at this scale.
